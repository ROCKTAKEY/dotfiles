# 開発におけるガイドライン
- t-wadaとKent BeckのTDDにより進めること。
  - テストの一環としてformatter、linter、型検査器なども実行すること。
  - 赤を確認し、いち早く緑となるよう実装し、そのあと必ずリファクタリングを行うこと。
  - 緑になったタイミング、及びリファクタリング後には必ずコミットを行うこと。
- 「Parse, don't validate」の思想に従い、データは速やかに検証し、検証した事実を型として保持すること。
- 型状態パターンを用い、状態遷移は型で表すことで、不整合な呼び出しが論理的に不可能なようにすること。
- 「Clean Architecture 達人に学ぶソフトウェアの構造と設計」やドメイン駆動開発の考え方に基づいて実装を行うこと。
  - SOLIDに従った設計を行い、実装の詳細から重要なビジネスロジックに向かって依存する。
  - 安定度と抽象度を一致させること。
  - 目的に沿った、システムの目的を叫ぶアーキテクチャを構成すること。
- Package by featureで構成すること。
- 指示のない限り後方互換性を考慮した実装は行わず、最適な実装を行うこと。
